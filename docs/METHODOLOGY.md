# 推荐方法说明

## 数据模式与发布边界

项目把“推荐排序”和“数据发布”分开。`APP_DATA_MODE=demo` 只读取仓库内的 `data/programs.demo.csv`；`APP_DATA_MODE=published` 只读取 SQLite 的 `published_programs` 活跃记录。公告正文、抽取 facts、待审核 draft、驳回 draft 都不属于推荐数据源。published 模式为空时返回空推荐列表，并显示“当前没有已审核发布的真实院校项目数据。”；非法模式值直接阻止应用启动。

正式项目必须经过：公告入库、字段提取、draft 人工审核、发布。系统保留公告 URL、标题、日期、内容哈希、证据等级、样本量、审核人和审核时间，便于复核。自动提取不能填出的字段保持缺失，发布前由审核员补齐；敏感名单公告不生成项目 draft，并记录跳过原因。

“发布”不等于“缺失字段已被猜测”：审核员可以发布字段仍未知但来源真实的项目，`missing_fields` 会进入推荐结果，未知门槛按高风险处理并降低置信度，绝不当作无门槛或低门槛。

## 1. 目标

当前模型的目标是构造一份合理的申请组合，不是预测个人是否必然录取。

## 2. 申请者强度

申请者强度范围为 0–100，主要权重：

- 专业排名：30；
- 本科院校画像：20；
- 科研经历：14；
- GPA：10；
- 英语：8；
- 论文成果：7；
- 竞赛：6；
- 工程项目：5。

这是 MVP 规则，不是经过真实案例拟合的统计结论。

## 3. 项目匹配分

项目匹配考虑：

- 本科院校画像差距；
- 专业排名是否达到基础线和偏好线；
- 科研与竞赛画像；
- 英语硬条件；
- 研究方向重合；
- 地区偏好；
- 培养类型偏好；
- GPA、论文与项目经历的细粒度加分。

## 4. 冲稳保分类

根据申请者强度减去项目难度参数的裕量分类。用户的风险偏好会调整分界线：

- 保守：更容易把项目归入冲刺；
- 均衡：默认分界；
- 进取：允许更高比例的高难度项目。

## 5. 置信度

置信度由以下因素决定：

- 证据等级；
- 匿名案例样本量；
- 数据距当前年份的时间；
- 是否是演示数据。

演示数据会受到强制降权。

## 6. 上线真实模型前的最低要求

- 每个项目至少覆盖两到三个申请年度；
- 明确区分夏令营、预推免和九推；
- 区分入营、优营、待录取和最终录取；
- 训练集与测试集按年份切分，避免未来信息泄漏；
- 分别评估冲刺、稳妥、保底分组的实际结果；
- 检查不同本科层次和专业方向上的系统性偏差；
- 公布数据缺失率和模型校准误差。
